# FinPulse — The Lending Echosystem

## Project Overview
Demo lending management system (LMS) for FinPulse. Handles the full loan lifecycle from application to closure.

## Tech Stack
- **Java 21**, Spring Boot 3.5.x, Maven
- **MySQL 8.0** — database name: `money_moment`
- **Spring Security + JWT** (JJWT 0.12.6, 24hr expiry, HMAC SHA256)
- **Spring Data JPA** with Hibernate
- **Lombok** — used extensively, do not remove annotations

## Running Locally
```bash
# Start MySQL first, then:
./mvnw spring-boot:run

# Or build and run jar:
./mvnw clean package -DskipTests
java -jar target/*.jar
```
- Dev port: **1992**
- DB: `localhost:3306/money_moment`, user `root`
- Prod profile (Docker/Render): port 8080, env vars `DB_URL`, `DB_USERNAME`, `DB_PASSWORD`, `JWT_SECRET`

## Base Package
`com.moneymoment.lending`

## Project Structure
```
controllers/    # REST controllers
services/       # Business logic
repos/          # Spring Data JPA repositories
entities/       # JPA entities
dtos/           # Request/Response DTOs
common/
  enums/        # Enums (LoanStatusEnums, EmploymentType, etc.)
  exception/    # Custom exceptions + GlobalExceptionHandler
  response/     # ApiResponse<T>, PagedResponse<T>
  spec/         # JPA Specifications (LoanSpecification, CustomerSpecification)
  utils/        # EmiCalculator, NumberGenerator, ProcessingChargeUtils, DateUtils
  validation/   # AadhaarValidator, PanValidator, PhoneValidator
security/       # JwtFilter, JwtUtil, SecurityConfig
master/         # Master data (loan types, rates, fees, statuses, etc.)
crons/          # EodScheduler (runs daily at 11:59 PM IST)
```

## Key Conventions
- All API responses wrapped in `ApiResponse<T>` — use `ApiResponse.success(data, message)`
- Paginated list responses use `PagedResponse<T>` inside `ApiResponse`
- DTOs follow `*RequestDto` / `*ResponseDto` naming
- Repositories extend `JpaRepository` + `JpaSpecificationExecutor` for filtered endpoints
- Custom exceptions: `ResourceNotFoundException`, `BusinessLogicException`, `DuplicateRecordException`, `ValidationException`
- `GlobalExceptionHandler` handles all exceptions centrally — do not add try/catch in controllers
- `BaseEntity` has audit fields; `CustomerEntity` and `LoanEntity` manage their own audit fields via `@PrePersist` / `@PreUpdate`

## Auth & Security
- JWT required on all endpoints except: `/api/auth/**`, `/api/health`, `/api/users/**`
- Header: `Authorization: Bearer <token>`
- Login: `POST /api/auth/login`

## Loan Lifecycle (status codes)
`INITIATED` → `UNDER_ASSESSMENT` → `APPROVED` → `DISBURSED` → `ACTIVE` → `OVERDUE` / `NPA` → `CLOSED`

## Important Business Rules
- Credit score is static (758.5) — demo project, no bureau integration
- EMI calculated using reducing balance method (`EmiCalculator`)
- Processing fee calculated via `ProcessingChargeUtils` using `ProcessingFeeConfigEntity`
- EOD scheduler (`EodScheduler`) updates DPD, overdue EMIs, and NPA flags daily
- Customers use **soft delete** — `DELETE /api/customers/{id}` sets `isActive = false`, never hard deletes
- Loan numbers generated by `NumberGenerator.generateLoanNumber()`
- Customer numbers generated by `NumberGenerator.generateCustomerNumber()`

## DDL
`spring.jpa.hibernate.ddl-auto=update` in both dev and prod — schema changes apply automatically on startup. No migration scripts needed.

## What NOT to Do
- Do not hard delete any financial records (loans, payments, penalties, EMI schedules)
- Do not skip `ApiResponse<T>` wrapper — all endpoints must use it
- Do not add `@Autowired` — constructor injection is used throughout
- Do not add Swagger/OpenAPI unless explicitly asked
- Do not push DB credentials — `application.properties` has local creds, do not commit changes to it
